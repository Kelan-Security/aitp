01

Cryptographic Identity

Every node generates a unique Ed25519 keypair. The EntityID is the stable identifier used for ACLs and trust scores.

Identity is the foundation. Before any AITP handshake, every participant must have cryptographic proof of who they are. We use Ed25519 — fast, secure, and post-quantum resistant.

The EntityID is derived as a SHA-256 hash of the public key, creating a deterministic, collision-resistant identifier for ACLs, audit trails, and trust scoring.

Key Derivation
EntityID = SHA-256(Ed25519_PublicKey)
struct EntityIdentity
struct EntityIdentity { id: [u8; 32], // SHA-256 hash public_key: Vec<u8>, metadata: HashMap<String, String>, }
02

Explicit Intent Declaration

Clients must declare their intent before the pipe opens. Unknown intents are dropped at the protocol layer.

Intent is the "why" of a connection. In legacy protocols, a server sees bytes on a port but has no idea what the client intends to do. AITP changes this.

Every client must cryptographically sign and send an explicit IntentCode during the handshake. The server validates this signature and only permits intents in its policy ACL. Rogue agents cannot fake intent.

enum IntentCode
enum IntentCode { ModelInference = 0x01, // LLM inference calls DataSync = 0x02, // Database replication ControlSignal = 0x03, // Management commands IdentityEx = 0x04, // Key exchange AuditLog = 0x05, // Log ingestion Heartbeat = 0x06, // Keep-alive Discovery = 0x07, // Service discovery }
03

Dynamic Trust Evaluation

Trust is computed in real-time from rules and AI analysis. A session is only granted if the merged score exceeds the threshold.

Trust is not binary. AITP uses a dual-scoring system: rules-based (fast, deterministic) and AI-driven (behavioral analysis via Ollama).

The final trust score is a weighted merge: S_merged = 0.4 × S_rules + 0.6 × S_AI

Scores are checked against fixed thresholds: Allow (≥128), Monitor (64–127), Deny (<64).

Trust Calculation
S_merged = 0.4 × S_rules + 0.6 × S_AI
Allow ≥ 128
Monitor 64 – 127
Deny < 64
Trust Thresholds
// Trust Decision Thresholds Allow: score ≥ 128 // Green light Monitor: score ∈ [64, 127] // Yellow flag Deny: score < 64 // Drop packet
04

Session Verdict & Grant

The server renders a verdict (Allow/Deny/Monitor) and cryptographically binds the session to the EntityID if approved.

Once trust evaluation is complete, the server issues a verdict and binds it to the session using HMAC-SHA256. This binding is sent to the eBPF enforcement layer.

If the verdict is Allow, the session is registered with a TTL (time-to-live) and a revocation nonce. If Deny, the packet is dropped and the client is added to a transient blocklist.

05

eBPF Wire-Speed Enforcement

Verdicts are enforced at the kernel layer via eBPF/XDP. Denied packets are dropped in <1μs, with sub-millisecond revocation.

The final primitive is enforcement. AITP doesn't just evaluate trust—it *implements* it at the network layer using eBPF (extended Berkeley Packet Filter).

Every packet is checked against the active session map. If the packet matches a denied session or an unknown entity, it is dropped at XDP (eXpress Data Path) — before it even reaches user-space code. This is why AITP achieves <1μs revocation latency.

Session revocation is also instant: the control plane sends a single command to remove the entry from the eBPF map, and all matching packets are immediately dropped.

The Vision

Why we are building AITP.

TCP/IP was designed for a world of trusted hosts and static routes. It cannot see the "Intent" of an LLM agent requesting access to a sensitive API.

AITP provides the missing metadata layer. By moving trust evaluation from the application to the transport layer, we eliminate entire classes of injection and lateral movement attacks before they reach your code.
